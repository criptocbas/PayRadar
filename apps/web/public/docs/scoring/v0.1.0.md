# PayRadar Scoring Engine v0.1.0

**Status:** Public, frozen.
**License:** MIT (`@payradar/scoring-engine`)
**Source:** `packages/scoring-engine/`
**Replay contract:** any historical score whose `engine_version = 0.1.0` is byte-for-byte reproducible from the same evidence.

This document is **load-bearing**: every signed score's `explain` link points here. If you change the formula, bump the version and publish a new `vX.Y.Z.md`. Old versions stay valid forever.

---

## 1. Contract

A score is the output of:

```
score = aggregate({
  reliability: reliabilityScore(probes, now),
  latency:     latencyScore(probes, peerBaselineP95, now),
  freshness:   freshnessScore(lastProbeTs, lastSeenInCatalog, now),
})
```

where each dimension function is **pure** (no I/O, no randomness, no implicit clock — `now` is passed in) and returns:

```ts
{
  score:           number 0..100,
  confidence:      number 0..1,
  weight:          number,
  evidence_count:  integer,
  version:         "0.1.0",
}
```

All weights, half-lives, and thresholds are constants in the source. **The aggregator weights each dimension by `weight × confidence`**, so under-evidenced dimensions can't move the score.

Default weights (sum to 1.0):

| Dimension    | Weight |
|--------------|--------|
| reliability  | 0.60   |
| latency      | 0.30   |
| freshness    | 0.10   |

Agents may override per-call (`?weights=reliability:0.5,latency:0.5`) — the canonical (default-weighted) score is always returned alongside.

## 2. Reliability

> *"Is it actually up?"*

Inputs: probes from the last 30 days for this endpoint.

Weighted by probe type:

| `probe_type`        | weight |
|---------------------|--------|
| `telemetry`         | 1.0    |
| `synthetic_paid`    | 0.9    |
| `liveness`          | 0.5    |
| `security`          | 0      |

Time decay: half-life = **7 days**. A probe from a week ago counts half as much as a probe from now.

Score: **Wilson lower bound** of weighted success rate at z = 1.96 (95% CI), times 100.

```
p     = sum(w_i * 1[ok_i]) / sum(w_i)
n     = sum(w_i)
z     = 1.96
denom = 1 + z²/n
center = (p + z²/(2n)) / denom
margin = z·sqrt(p(1-p)/n + z²/(4n²)) / denom
score = max(0, center − margin) × 100
```

Confidence: `min(1, log10(probe_count + 1) / 2.5)` — saturates around 316 probes.

Cold start (zero probes): score = 50, confidence = 0.

## 3. Latency

> *"How fast vs. peers?"*

Inputs: latencies from successful probes in the last 30 days.

Time decay: half-life = **24 hours**.

Peer baseline (p95, ms): median of peer-cohort p95s.

- A "peer" is any other endpoint sharing at least one capability tag.
- A peer needs **≥ 5 successful samples** to count; a cohort needs **≥ 2 peers**.
- If no cohort qualifies → fall back to global default (1500 ms).

Score:

```
ratio       = endpoint_p95 / peer_baseline_p95
base        = clamp(0, 100, 100 × (2 − ratio))
tail_ratio  = endpoint_p99 / endpoint_p50
tail_bonus  = clamp(0, 10, 10 − (tail_ratio − 1) × 5)
score       = min(100, base + tail_bonus)
```

Reading: 100 means "p95 ≤ ½ peer baseline." 0 means "p95 ≥ 2× peer baseline."

Confidence: `min(1, log10(samples + 1) / 2)` — saturates around 100 samples.

Future upgrade: pgvector capability embeddings expand the peer set from exact-tag to `cosine_sim > 0.85`.

## 4. Freshness

> *"How current is our evidence?"*

```
probe_freshness    = 100 × 0.5^(hours_since_last_probe / 6)        # half-life 6h
catalog_freshness  = 100 × 0.5^(hours_since_last_catalog / 24)      # half-life 24h
score              = 0.7 × probe_freshness + 0.3 × catalog_freshness
```

Confidence: 1.0 if a probe exists, else 0.5.

Why this exists: an endpoint last probed two weeks ago might score 99 on reliability and 95 on latency, but the score still represents stale evidence. Freshness provides the "this needs refreshing" signal that lets agents and operators tell apart "great endpoint" from "great endpoint *as of when we last checked*."

## 5. Aggregation

```ts
let weightedScoreNumerator     = 0
let weightedConfidenceNumerator = 0
let totalEffectiveWeight       = 0
let totalNominalWeight         = 0
let totalEvidence              = 0

for (name, dim) of dimensions:
  w         = weightOverrides[name] ?? DEFAULT_WEIGHTS[name] ?? dim.weight
  effective = w × dim.confidence

  weightedScoreNumerator      += dim.score      × effective
  weightedConfidenceNumerator += dim.confidence × w
  totalEffectiveWeight        += effective
  totalNominalWeight          += w
  totalEvidence               += dim.evidence_count

score      = weightedScoreNumerator      / totalEffectiveWeight
confidence = weightedConfidenceNumerator / totalNominalWeight
```

**Cold-start cap.** If `totalEvidence < 50`, score is capped at 65 and tier is `PROVISIONAL`, regardless of computed value.

### Tiers

| Tier         | Score range |
|--------------|-------------|
| `S`          | ≥ 92        |
| `A`          | 80 – 91.9   |
| `B`          | 65 – 79.9   |
| `C`          | 50 – 64.9   |
| `D`          | 30 – 49.9   |
| `F`          | < 30        |
| `PROVISIONAL`| insufficient evidence |

## 6. Signing

Every `Score` is canonicalized (sorted-key JSON, no whitespace) over the load-bearing fields:

```ts
canonicalize({
  endpoint_id, computed_at, engine_version,
  score, confidence, tier, dimensions
})
```

then signed with **ed25519**. The signature is stored alongside the score and returned via `/v1/discover`. The public key is published at:

```
GET /.well-known/payradar-keys.json
```

Verifiers fetch the key matching `score.signature.key_id`, then verify against the canonical bytes.

`score_id` is intentionally **not** signed. The same evidence produces the same signature, which makes replay verification straightforward.

## 7. Replay

To reproduce any historical score:

1. Fetch the probe set used (`/v1/score/{score_id}/replay` — coming in v0.2; today: query `probes` since `score.computed_at - 30d`).
2. Pin `@payradar/scoring-engine` to the matching `engine_version`.
3. Call the same dimension functions with `now = score.computed_at`.

Output is byte-for-byte deterministic.

---

## 8. Versioning

- Bug fix in a dimension formula → new dimension `VERSION` *and* new engine `ENGINE_VERSION`.
- Default-weight tweak → new `ENGINE_VERSION` only.
- Stub dimension goes live (e.g., `security` ships) → new `ENGINE_VERSION`, dimension `VERSION = 0.1.0`.

The engine versions of `reliability`, `latency`, `freshness`, and the aggregator are independent and surfaced in every score.

## 9. Known limitations (v0.1)

- Six of the nine spec'd dimensions (trust, signal, security, pricing-value, community-adoption, on-chain-reputation) are **stubbed at zero weight** — they don't appear in `dimensions`. Roadmap: roll them in over v0.2 → v0.5.
- Liveness probes from a single region. Geo-distributed probes coming in v0.3.
- Capability-cohort peer matching uses exact tags; pgvector-based semantic matching is the upgrade path.
- Telemetry and synthetic-paid probes are weighted higher than liveness, but the v0.1 ingestor only emits liveness — so all current evidence is at the lowest weight tier. This is honest but conservative; expect scores to firm up as paid probes ship in v0.2.
